Q1How should a live session work on the Mac (normal repo and game repo) — start and finish?

Start: open the session in the project checkout. On first contact with an area (question or edit), read .docs/index.md → the area file. About to edit → branch off base first. Working backlog items → the backlog skill governs pick/execute/update.

  • Normal repo: a branch in the checkout is always factory-safe — the driver never touches your checkout. A sibling worktree is optional convenience for parallel work.
  • Game repo: same, except never a worktree (Unity pins one project path; Godot gains nothing and pays a reimport). In-checkout branch, one at a time. Engine skills add mechanics: close the Unity editor before batchmode; run godot --headless --import before headless work.

Finish: end pushed — merged or as a PR (the merge gate handles green factory PRs). Update touched .docs/ area files, .docs/known-issues.md entries you hit or resolved, and backlog task states as part of the work. prep only if you left the checkout dirty and want it reset — it is repair, not a ritual.

Q2Same, but on the VPS?

Identical contract — the machine changes nothing. Two VPS-specific caveats: some sites (e.g. code4food) are served straight from the checkout, so uncommitted edits show on the live site and a prep would revert them — commit promptly; and in practice VPS sessions start through rc (Q3) rather than raw ssh.

Q3How should an rc session work on the VPS — and does it need to finish?

  • Start: rc <folder>. Git repos are auto-served through the pilot worktree ~/pilot/<name> (branch pilot/<name> from origin base) inside tmux rc-<name>; it prints the claude.ai link (rc -u re-prints, rc -l lists).
  • Land work: tell the session "commit this and open a PR" — the gate does the rest.
  • Does it need to finish? No. The tmux server can stay idle indefinitely; it is designed to survive disconnects, and a clean pilot is fast-forwarded to origin on the next rc run. When you want it gone: rc -k <folder> (graceful Ctrl-C first — a hard tmux kill-session is what left a factory blocked by locked worktrees on 2026-07-14). tmux does not survive reboot; just re-run rc.

The pilot worktree is convenience + kill-isolation, not a safety requirement — the driver never resets checkouts.

Q4How should Eva monitor a factory window?

Per the monitor contract (FACTORY.md §Architecture & contracts):

  • Read (what → status → why → work-product): escalations outbox → GET /api/state → <state>/log/dev-*.out → gh.
  • Act, escalating: gh-level (answer needs-human issues, merge green PRs if authorized) → fixes on her own branches pushed to origin → control-plane (dashboard POSTs, STOP, supervisor keep/release) → escalate to the owner (Telegram/voice).
  • Never: enter ~/.factory/worktrees/, kill processes (the supervisor's job), or raw-git in a project checkout.

Cross-machine: Eva reads the VPS outbox over ssh — transport is the consumer's job by design.

Q5How should anything monitor a factory window?

Exactly the same — the contract is deliberately consumer-agnostic: the interface is files + token'd HTTP + gh. Any agent, script, or human that can read JSONL, hit the dashboard, and run gh is a valid Layer 3. Eva is the reference implementation, not a requirement. What a monitor may merge is a per-owner parameter, never a product default.

Q6How do projects find the factory rules, and where are they?

Authority lives machine-side at ~/.factory/runtime/factory/FACTORY.md §Architecture & contracts (source: factory/FACTORY.md in the dev repo; shipped via publish → deploy-runtime to every machine). Routing: the dev repo's .docs/index.md topic router points at it, and the backlog skill — loaded by every session inside a factory project — opens with a pointer to the same section. This page and the full guide are served by every dashboard.

Q7How are docs maintained now, and what docs exist?

Rules (docs skill 1.4.0 + installed CLAUDE.md block): read the index before answering or touching; update touched area files inline after small/feature work; reader-repair contradictions on the spot; check .docs/known-issues.md before debugging and delete its entries in the change that fixes them; human docs updated in the same branch; contract changes update FACTORY.md §contracts in the same PR; authority order source code > .docs/ > memories/chat. No length quotas — area files scale by splitting areas; every bullet must earn its place.
