Overview

Claude Code develops a fully-specced product alone in scheduled windows: a fresh headless session per task, state carried in files, humans feeding input async. The same driver runs macOS, Linux/VPS, and Windows. State splits in two:

  • In the repo (travels via git): .factory/spec, .factory/backlog/, .factory/inbox/, .docs/ — the collaboration surface.
  • Machine-side (never in git): ~/.factory/projects/<name-hash>/ ("<state>") — config, schedule, logs, STOP; ~/.factory/runtime/ — the driver, advanced only by deploy-runtime; ~/.factory/registry.json — this machine's factories.

The one invariant

Origin is the rendezvous point. Factory sessions start from origin/<base> in throwaway worktrees under ~/.factory/worktrees/ — never from local state. Work becomes visible to the factory, to other machines, and to other sessions only when pushed.

Everything follows: live sessions any time, two machines on one project (time-shifted), every repo type — the contract is branch → push/PR → converge at origin, which never requires a worktree of the owner, so no engine restriction breaks it.

 OWNER (live session)                      FACTORY (window)
 checkout, own branch                      ~/.factory/worktrees/, per-session
        │                                          │
        │ commit → push / PR                       │ task branch → PR
        ▼                                          ▼
        └────────────►   origin/<base>   ◄─────────┘
                      (the ONLY meeting point)
                               │
             merge gate merges green PRs; backlog flips ride along
                               │
                               ▼
        next session / next machine / owner pulls — everyone converges

The owner's checkout belongs to the owner: the driver never flips its branch, never quarantines WIP. Its single touch: fast-forward when the checkout is clean AND on base AND strictly behind. prep is the explicit repair verb for a dirty checkout — not a required handoff.

The three layers

LayerWhatDuties
1 · Driverfactory.mjsruns windows: spawn sessions, enforce caps, gate merges, self-heal (breakers, turn-cap handoffs, needs-human issues)
2 · Supervisorsupervisor.mjs, one per machinekill hung runs past wall-clock bound (then prep), owner-directed relaunches, escalate wedged factories (factory-stuck after repeated dead scheduled windows), write the escalations outbox; read-only otherwise
3 · MonitorEva / any agent / live session / humanoptional + replaceable; consumes the read surfaces, acts through the sanctioned verbs (contract below)

How a window runs

 schedule fires (or ▶ on the dashboard, or `factory.mjs dev`)
   │
   ▼
 triage (1 session)  — inbox + issues → backlog updates + plan of day
   │
   ▼
 dev loop: ┌─ pick next actionable task (plan.json / backlog)
           │  fresh worktree from origin/<base>  — tooling injected from runtime
           │  implement (TDD + verify) → PR → report_status
           │  driver: gate polls checks → merges green locally → push
           └─ next fresh session … until window end / STOP / cap / no tasks
   │
   ▼
 window-end sweep — leftover green PRs get one gate pass
   │
   ▼
 report (1 session) → daily-log issue + mirrors — honest summary
  • A window with zero actionable tasks skips itself before spawning a paid session.
  • Budget = windowHours × maxSessionsPerWindow × maxTurnsPerSession × sessionTimeoutMin — those four ARE the cap.
  • Parking states: blocked (machine-clearable, triage re-opens) vs needs-human (only the owner clears). Only-needs-human left → status "waiting on owner"; only dependency-blocked left → "deadlocked".

Live sessions (piloting)

First-class, any time, on any machine — the factory is built to interleave with them.

Start

  1. Open the session in the project checkout.
  2. First contact with an area — question or edit — read .docs/index.md, then the area file.
  3. Editing → branch off base first. Backlog work → the backlog skill.

Where to work, by repo type

RepoWhereWhy
normal (webapp, api, general)in-checkout branch; sibling worktree optionalalways factory-safe — the driver never touches the checkout
Unityin-checkout branch, one at a time — never worktreeseditor pins one project path; dot-dir ancestors break .NET config
Godotin-checkout branchworktree buys nothing, costs a reimport
iOS/Androidin-checkout branch; worktree if you liketoolchain-tolerant — owner's choice

Finish

  • End pushed — merged or as a PR; the merge gate handles green factory PRs.
  • Update touched .docs/ area files, .docs/known-issues.md entries you hit or resolved, and backlog task states as part of the work.
  • prep only if the checkout was left dirty/diverged (it quarantines strays to <state>/log/quarantine-*/ + stash, resets to origin base, pushes unpushed commits).

VPS extra: checkout-served sites (code4food) show uncommitted edits live, and prep would revert them — commit promptly. Don't manually start a window on a project you're mid-piloting in the checkout of.

rc sessions (VPS, phone)

rc is remote-access ergonomics (tmux + Claude Remote Control), not a factory feature. rc <folder> serves a git repo through a pilot worktree ~/pilot/<name> (branch pilot/<name>) in tmux rc-<name> and prints the claude.ai link. The pilot worktree is convenience + kill-isolation, not a safety requirement.

  • rc -l list · rc -u <folder> re-print link · rc -k <folder> graceful stop.
  • Land work by telling the session: "commit this and open a PR".
  • The tmux server may stay idle — a clean pilot fast-forwards to origin on the next rc. Never hard-kill the tmux session (locked-worktree risk); tmux doesn't survive reboot, just re-run rc.

Monitoring a window (Eva or anything)

Consumer-agnostic by design: the interface is files + token'd HTTP + gh. Eva is the reference implementation (phased plan: relay → watcher → resolver), not a requirement.

Read surfaces — what → status → why → work-product

  1. ~/.factory/escalations.jsonl — append-only outbox, one JSON/line (ts, machine, project, name, type, detail); consumers track their own offset; cross-machine transport is the consumer's job.
  2. GET /api/state — the whole fleet as JSON; GET /api/log for driver logs.
  3. <state>/log/dev-*.out — full session JSONL (grep "type":"result", report_status, open_question).
  4. gh — PRs, checks, needs-human issues, the daily-log issue.

Act ladder — least to most invasive

  1. gh-level: answer needs-human issues, comment on PRs, merge green PRs if authorized (per-owner parameter, never a default).
  2. repo-level: fixes on the monitor's OWN branches pushed to origin — a monitor is just another live session.
  3. control-plane: dashboard POST /api/run|stop|resume|enabled, touch <state>/STOP, supervisor keep/release.
  4. escalate to the human — with the record and a suggested action.
Never: enter ~/.factory/worktrees/ (the window's territory) · kill processes (Layer 1 does it safely, then preps) · raw git in a project checkout · edit <state> internals.

Where the rules live

  • Authority: ~/.factory/runtime/factory/FACTORY.md §Architecture & contracts (source: factory/FACTORY.md in the dev repo; shipped by publish → deploy-runtime, identical on every machine). A PR that changes a contract updates that section in the same PR.
  • Routing: the dev repo's .docs/index.md topic router; the backlog skill (loaded in every factory project) opens with a pointer to the contracts; this page is served by every dashboard at /docs.
  • History: factory/NOTES.md + factory/specs/* — banner'd design records, never current contract.

How docs are maintained

  • Read .docs/index.md before answering about or touching an unfamiliar area — verifying a feature exists is cheaper than advising from memory of it.
  • Authority order: source code > .docs//project docs > memories/summaries/chat. Memories are leads to verify, never authorities.
  • Update touched area files inline at the end of small/feature work; reader-repair contradictions on the spot; human docs (README/ONBOARDING/CLAUDE.md) updated in the same branch.
  • .docs/known-issues.md — living list of open bugs, quirks, and workarounds, standard in every project (skillset 1.4.0): check it before debugging; an entry is deleted in the same change that fixes it.
  • Machine memories hold machine facts and dated journals only; fleet-wide doctrine goes in the repo, where it travels through origin like code.
