# Spec — two-way GitHub Projects sync (NOTES item 20)

> Design record, NOT current contract — on conflict, `FACTORY.md` and the
> driver source win.

STATUS: BUILT and ACTIVE (2026-07-06) on all three VPS factories — kept as
the design reference; as-built implementation notes are inline in the build
plan at the bottom.

Spec'd 2026-07-06 in a planning session with the user. Goal: keep the
whole factory visible and steerable from GitHub — "everything in one
place" — without moving the source of truth off disk.

## Principles

- **Backlog markdown stays the source of truth.** Sessions never read the
  board; only the driver's sync step touches it. If GitHub is down, the
  factory runs exactly as today.
- **Deterministic, driver-side, zero model tokens.** Sync is plain code
  (like the merge-gate), not a session prompt. Runs from call sites that
  already exist in the dev loop.
- **Inbound goes through the inbox.** Human board edits become a note in
  `.factory/inbox/` that triage folds in with its existing rules — no new
  triage machinery, and human input is judged by a model, never
  auto-applied to the backlog.

## Config & state

- Opt-in per factory: `"board": {"github": true}` in `.factory/config.json`.
- `.factory/board.json` (gitignored): cache written by `--init` —
  project number, project node ID, Status field ID + option IDs, Epic
  field ID, and a `taskId → item node ID` map maintained by every sync.
- Auth: the existing gh login, PLUS the `project` scope —
  `gh auth refresh -s project` (same blocker class as the 2026-07-04
  workflow-scope incident; `factory doctor` item 21 checks it when board
  is enabled).

## CLI

```
node factory.mjs sync-board --project <path>          # one sync pass
node factory.mjs sync-board --project <path> --init   # create board + fields, write board.json
```

`--init` creates a Projects v2 board named after the factory (owner =
repo owner, linked to the repo), adds a `Status` single-select with
options exactly matching the backlog vocab — `todo`, `in-progress`,
`review`, `blocked`, `done` — and an `Epic` single-select (options from
`.factory/backlog/*.md` filenames). Re-running `--init` is idempotent
(finds the existing board by name).

## Outbound (backlog → board)

One pass:

1. Parse `.factory/backlog/*.md` (same parser shape as the dashboard's,
   dashboard.mjs `readBacklog`): task id, title, status, epic, PR links.
2. For each task not in board.json's map: `item-add` (title:
   `T-012 — <title>`), record node ID.
3. For each task whose board Status ≠ backlog status: `item-edit` the
   Status field. Set Epic on first add. Put PR link + Model/Effort hints
   in the item body (updated when they change).
4. Tasks deleted from the backlog: archive the board item (don't delete —
   preserves history).

Call sites in the dev loop (all already exist as code points):
window start (right after the plan is loaded), after each merge-gate
merge, window end. Plus the triage and report wrappers run one pass after
their session ends. 4–6 passes/day, far under GraphQL rate limits.

Failures follow the notify rule: log and continue, never fatal, never
arm the breaker.

## Inbound (board → inbox)

In the same pass, after pushing:

1. Fetch all board items (single GraphQL query, paginated).
2. **New items** not matching any backlog task id (drafts the user typed
   on the board, or issues someone added): collect title + body.
3. **Status mismatches in the human direction** — board Status differs
   from backlog after our own push (i.e. a human moved a card since the
   last sync; detected by comparing against the status we just wrote).
4. If anything was collected, write/overwrite
   `.factory/inbox/board-delta.md`:

   ```markdown
   # Board delta (generated by sync-board, <ISO timestamp>)
   ## New items on the board (add to backlog or reject with a reason)
   - "<title>" — <body excerpt>
   ## Human moves (judge each; the backlog is authoritative for factory-made status)
   - T-014: board says `todo`, backlog says `done` (merged 2026-07-06) — probably a re-open request?
   ```

5. Triage already processes every inbox file and deletes it (triage.md
   step 1). Add one line to triage.md naming this file: new items → new
   tasks per the backlog skill; contradictory moves → judge, and when in
   doubt file needs-human rather than guess. After triage commits backlog
   changes, the next sync pass makes the board consistent again.

## Conflict rules

- **Factory wins on status.** The board lagging (task moved to `done` by
  the gate while the card sits in `review`) is just staleness; outbound
  overwrites it. A human dragging a card AGAINST factory state (merged
  task back to `todo`) is never auto-applied — it lands in the delta note
  for triage to interpret (usually: a re-open request → new bug task).
- **Human wins on new items and priority.** New cards become tasks via
  triage (which may also reject with a reason in the daily log). Column
  order / manual ranking on the board is advisory input to triage's
  plan.json ordering, not a hard order.

## Non-goals (v1)

- No per-turn or webhook-driven sync — polling at the existing call
  sites is enough at factory cadence.
- No sync of spec/decision documents; issues and PRs already live on
  GitHub natively.
- No multi-repo boards; one board per factory.

## Build plan (two sessions)

1. **Outbound**: `--init` + push pass + dev-loop/triage/report call
   sites + FACTORY.md section + doctor scope check. Verify: run against a
   real factory, eyeball the board, flip a backlog status, re-run, watch
   the card move.
   *Built 2026-07-06 — implementation notes:*
   - *Lives INSIDE factory.mjs: projects run a single-file driver copy
     (`.factory/driver.mjs`, made by init), so no second module.*
   - *Uses `gh project` subcommands (item-create/item-list/item-edit/
     item-archive/field-list/field-create) + two raw GraphQL calls:
     replacing the built-in Status options with the backlog vocabulary at
     init, and fetching DraftIssue content ids (lazily, only when a body
     changed — item bodies carry Model/Effort + PR links and are updated
     on change).*
   - *Sync points: window start, after EACH dev session (sessions commit
     the status flips), window end, after triage. Diffs against
     `item-list` so an unchanged backlog makes zero write calls.*
   - *v1 limitation: Epic single-select options are created at --init
     from the then-existing epic files; tasks in epics added later leave
     Epic unset (status still syncs). Re-run --init after adding epics.*
   - *Verified against a stateful stub gh: init, create+move+epic, zero-
     write idempotency, body update via draft ids, archive on task
     removal, failure isolation (gh dying mid-sync logs and continues).
     Live-board verification pends `gh auth refresh -s project`.*
2. **Inbound**: fetch/diff + board-delta.md writer + triage.md line.
   Verify: add a draft card + drag a done task to todo, run sync, read
   the generated delta, run a triage, confirm the fold.
   *Built 2026-07-06 — implementation notes (differ from the sketch above
   in two ways):*
   - *Human-move detection uses last-pushed state, not a separate diff
     pass: board.json's items map stores `{itemId, status}` where status
     is what the driver last wrote (v1 bare-string maps normalize on the
     fly). Board ≠ last-pushed AND ≠ backlog ⇒ human move → recorded in
     the delta, then the outbound write restores factory state in the
     same pass. A human move that AGREES with a pending backlog change is
     silently absorbed (no delta, no write).*
   - *New cards are captured into the delta (title + body) and then
     ARCHIVED immediately — otherwise the human's draft would duplicate
     the real task card triage creates from it. The delta note is the
     artifact of record, per inbox semantics (process, then delete).*
   - *Delta file appends across syncs until triage consumes it; moves
     can't re-report (after restore, board == last-pushed again) and
     archived newcomers can't re-report (item-list omits archived).*
   - *Stub-verified: move detected + restored, card captured + archived,
     next pass silent with zero writes. Live fold-through-triage pends
     the next scheduled triage.*
